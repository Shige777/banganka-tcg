# ERROR_HANDLING.md
> 万願果 エラーハンドリング・リカバリ仕様 v1.0

---

## 0. 目的

ゲーム中に発生しうるエラーの分類・表示・リカバリ方法を定義する。
ASSET_ACCOUNT_SETTINGS.md §5 のエラーUIアセットと連携する。

---

## 1. エラー分類

| カテゴリ | 深刻度 | ユーザー影響 |
|---------|--------|------------|
| A: 通信エラー | 高 | 対戦中断・データ未保存 |
| B: サーバーエラー | 高 | 機能利用不可 |
| C: クライアントエラー | 中 | 操作失敗 |
| D: メンテナンス | 低 | 一時的利用不可 |
| E: データ不整合 | 高 | 進行不能 |

---

## 2. カテゴリ別 エラーハンドリング

### 2.1 A: 通信エラー

| 状況 | 表示 | リカバリ |
|------|------|---------|
| 対戦中に通信断 | 「接続が切れました。再接続中...」+ スピナー | 自動リトライ（3回・2秒間隔）→ 復帰 or 切断画面 |
| 対戦外で通信断 | 「通信エラーが発生しました」+ リトライボタン | 手動リトライ。オフラインでも閲覧可能な画面はそのまま表示 |
| パック購入中に通信断 | 「購入処理中にエラーが発生しました。通信回復後に反映されます」 | サーバー側でトランザクション保証。再起動時にペンディング購入を確認・完了 |
| ルーム参加中に通信断 | 「接続に失敗しました」+ リトライボタン | リトライ or バトル画面に戻る |

**自動リトライ戦略**：
- 1回目：即時
- 2回目：2秒後
- 3回目：4秒後
- 3回失敗 → エラーパネル表示（手動リトライ or ホームに戻る）

### 2.2 B: サーバーエラー

| 状況 | 表示 | リカバリ |
|------|------|---------|
| Cloud Functions エラー（500） | 「サーバーエラーが発生しました」+ リトライボタン | リトライ。5分以上続く場合はメンテナンス扱い |
| Firebase Quota超過 | 「アクセスが集中しています。しばらくお待ちください」 | 30秒後にリトライ |
| 認証トークン期限切れ | — (バックグラウンドでリフレッシュ) | Firebase SDK が自動でトークンリフレッシュ。失敗時のみ再ログイン誘導 |

### 2.3 C: クライアントエラー

| 状況 | 表示 | リカバリ |
|------|------|---------|
| 不正操作（CP不足等） | UI側で操作不可にする（グレーアウト） | エラー表示なし。UI制御で防止 |
| 不正コマンド（サーバー拒否） | — (ログのみ) | クライアント側のStateを再同期 |
| アプリクラッシュ | — | 再起動時に対戦中なら自動復帰（NETWORK_SPEC.md §6.2） |

### 2.4 D: メンテナンス

| 状況 | 表示 | リカバリ |
|------|------|---------|
| 計画メンテナンス | 「メンテナンス中です（終了予定：XX:XX）」+ メンテアイコン | 終了予定時刻を表示。アプリは起動可能だが対戦・ショップ不可 |
| 緊急メンテナンス | 「緊急メンテナンス実施中です」 | 終了予定未定の場合は「復旧次第お知らせします」 |
| バージョン強制更新 | 「新しいバージョンが公開されています」+ App Storeボタン | App Storeへ誘導。更新完了まで対戦不可 |

**メンテナンスフラグ管理**：
- Firebase Remote Config に `maintenance_mode: boolean` と `maintenance_message: string` を設定
- アプリ起動時 + 定期チェック（5分間隔）で確認

### 2.5 E: データ不整合

| 状況 | 表示 | リカバリ |
|------|------|---------|
| ローカルとサーバーのカードデータ不一致 | — (バックグラウンド修正) | サーバーデータを正として上書き |
| 対戦中のGameState不整合 | — (ログ出力) | サーバーのGameStateで強制同期 |
| デッキデータ破損 | 「デッキデータの読み込みに失敗しました。初期デッキに戻します」 | スターターデッキにリセット |

## 3. エラー表示UI

### 3.1 表示パターン

| パターン | 使用場面 | アセット |
|---------|---------|---------|
| **フルスクリーン** | メンテナンス、強制更新 | ERROR_PANEL_BG + ERROR_ICON_MAINTENANCE |
| **モーダル** | 通信エラー、サーバーエラー | ERROR_PANEL_BG + ERROR_BTN_RETRY + ERROR_BTN_CLOSE |
| **トースト** | 軽微な通知、操作確認 | NOTIFY_POPUP_BG + NOTIFY_ICON_INFO |
| **インライン** | ルームID不正、入力バリデーション | テキストのみ（赤色） |

### 3.2 ボタン構成

| 状況 | ボタン |
|------|--------|
| リトライ可能 | 「もう一度試す」（プライマリ）+「戻る」（セカンダリ） |
| リトライ不可 | 「ホームに戻る」（プライマリ） |
| メンテナンス | 「閉じる」のみ |
| 強制更新 | 「App Storeを開く」（プライマリ） |

## 4. ログ・モニタリング

### 4.1 エラーログ（Cloud Functions）
- 全エラーを Cloud Logging に出力
- エラーレート閾値：5分間で100件超 → アラート

### 4.2 クライアントログ（Firebase Crashlytics）
- クラッシュレポート自動収集
- Non-fatal エラー（通信エラー等）もログ送信
- ユーザーID紐付けで問い合わせ対応可能

### 4.3 監視ダッシュボード
- Firebase Console で確認可能：
  - Crashlytics（クラッシュ率）
  - Performance Monitoring（レスポンス時間）
  - Cloud Functions ログ

## 5. 対戦中のエッジケース

| ケース | 処理 |
|--------|------|
| 両者が同時にコマンド送信 | Realtime Database のトランザクションで順序保証。先着優先 |
| ターン切替中にコマンド | activePlayerチェックで拒否 |
| 対戦終了後にコマンド | isGameOver チェックで拒否 |
| **HP=0 判定時に切断** | **サーバー側で KO ジャッジメント完了 → ファイナルステート発動 → 結果保存** |
| **ターン24終了時の HP 比較中に切断** | **サーバー側で塗り勝利判定完了 → 結果保存（切断側に敗北判定の場合も同様）** |
| デッキ枯渇（0枚でドロー） | ドロースキップ。手札0枚でもプレイ可能な行動があれば続行 |
| **複数 WishTrigger 連鎖中に切断** | **サーバー側でカスケード処理完了後に状態同期** |
| **HP ダメージ計算エラー** | **数値が 0～100 クランプ。異常値時はサーバー値で上書き** |

> WishTrigger連鎖中の切断・エラー処理は、発動キューを全て解決してからタイムアウト判定を行う。詳細は GAME_DESIGN.md §5.5 を参照。
