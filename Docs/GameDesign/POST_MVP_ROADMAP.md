# POST_MVP_ROADMAP.md
> 万願果 MVP後拡張ロードマップ v2.1

---

## 0. 目的

`万願果` のMVP完成後に何をどの順で拡張していくかを整理する。
AI や実装者が、MVP以降の開発を場当たり的に進めないための指針。

---

## 1. 前提

- `GAME_DESIGN.md` がゲームルールの正本
- MVP v3.0 のスコープ（PRODUCT_REQUIREMENTS.md §4 参照 + v0.5.4 昇格分）:
  - 基本対戦ループ（オンライン1v1 + AI Bot対戦）
  - 5画面シェル + デッキ構築UI
  - マネタイゼーション基盤（パック・バトルパス・広告・クラフト・ショップ）
  - チュートリアル
  - フレンド対戦
  - **ランクマッチ**（GAME_DESIGN.md §14）
  - **ドラフトモード**（GAME_DESIGN.md §15）
  - Firebase バックエンド + ネットワーク同期
- このファイルは、MVPの後に追加する拡張を段階的に定義する

> **v1.0 → v2.0 変更点**: MONETIZATION_DESIGN.md v2.0 の決定により、旧Phase 6「収益化と運営」の内容（パック・バトルパス・広告・通貨設計・pity・重複救済）はすべてMVPスコープに昇格済み。デッキ構築UIもMVP必須に変更。

---

## 2. ポストMVPの目的

MVP後は、以下を強化して正式サービス→成長フェーズに進む。

- コンテンツ量の拡充（カードプール・ストーリー）
- 対戦の競技性向上（ランク・シーズン）
- リテンション強化（イベント・ソーシャル）
- ストーリーの厚み
- プラットフォーム拡張（Steam → Android）
- 運営効率化

---

## 3. 開発フェーズ

### Phase 1: 完成度向上（MVP後 1〜2ヶ月）

目的: MVPを「ソフトローンチ品質」から「正式サービス品質」に引き上げる

実装候補:
- UI polish とアニメーション品質向上（ANIMATION_SPEC.md 完全準拠）
- サウンド演出強化（SOUND_DESIGN_SPEC.md のダイナミックBGM実装）
- アクセシビリティ P1 項目（カラーブラインドモード設定、テキストサイズ設定）
- VoiceOver基本対応
- パフォーマンス最適化（PERFORMANCE_SPEC.md 全項目クリア）
- 文言統一（CARD_TEXT_GUIDELINES.md 全カード適用）
- バグ修正とバランス初回パッチ

### Phase 2: カードゲームとしての厚み（MVP後 2〜4ヶ月）

目的: 対戦の幅とデッキ構築の面白さを増やす

> **カードリリースサイクル**: 「大弾＋ミニセット」2段階構成を採用。
> 詳細は `CARD_RELEASE_ROADMAP.md` を参照。
> - 大弾（年3回・40〜50種）: 新キーワード・新シナジー・ストーリー新章
> - ミニセット（年3回・10〜15種）: メタ補正・既存キーワード派生・弱願相底上げ
> - 初年度はMVP後2ヶ月で大弾①を前倒しリリース

実装候補:
- **大弾① 第2弾シリーズ**（新カード40〜50種）— MVP後2ヶ月目標
- **ミニセット①**（メタ補正カード10〜15種）— 大弾①の6週後
- 複数願主対応（6願主をすべてプレイ可能に）
- 願相別アーキタイプの強化（各願相2〜3種のデッキ型）
- 界律バリエーション追加
- カードプール拡張に伴うバランスパッチ（BALANCE_POLICY.md準拠）
- **構築制限イベント**開始（EVENT_SYSTEM_SPEC.md）
- **季節イベント**開始
- ドラフトイベント拡張（基本ドラフトはMVP済み。イベント限定ルール追加）
- SNSシェア（Twitter/LINE デッキコード・対戦結果）
- プレミアムエモート販売開始

#### Phase 2 補足: カードローテーション・フォーマット方針

**フォーマット種別（Phase 2以降で段階導入）:**

| フォーマット | 使用可能カード | 導入時期 |
|------------|-------------|---------|
| **スタンダード** | 直近3シリーズ + ベーシック | 第3弾リリース時 |
| **エターナル** | 全シリーズ（制限カードあり） | 第3弾リリース時 |
| **イベント限定** | ルール毎に変動 | Phase 2〜 |

**ローテーション運用:**
- 第3弾リリース時にスタンダード/エターナルの2フォーマット制を導入
- ローテーション周期: 新シリーズ追加時に最も古いシリーズがスタンダード落ち
- スタンダード落ちの事前告知: 2ヶ月前にアプリ内通知
- エターナルの制限リスト: 月次更新（BALANCE_POLICY.md の監視プロセスと連動）
- ランクマッチはスタンダード準拠、フレンド対戦は両フォーマット選択可

### Phase 3: 競技性とソーシャル（MVP後 4〜6ヶ月）

目的: 本格的な継続対戦ゲームにする

> ランクマッチ・ドラフトモードは v0.5.4 で MVP に昇格済み（GAME_DESIGN.md §14, §15）。

実装候補:
- **シーズン報酬拡張**（限定スリーブ・称号・フレーム）— ランクマッチ自体はMVP
- **リプレイ機能**（REPLAY_SPEC.md）
- **観戦モード**（2ターン遅延）
- 対戦履歴画面
- フレンド対戦カスタムルール
- プレイヤープロフィール拡張（戦績公開・称号）
- プッシュ通知本格運用（NOTIFICATION_SPEC.md）
- ランキングイベント

#### Phase 3 補足: ランクマッチ・シーズン設計方針

**ランクシステム:**
- 段位制: 銅 → 銀 → 金 → 白金 → 宝石 → 願主（各5段階、計30段）
- マッチメイキング: 同ランク帯±2段の範囲でペアリング
- レーティング: 内部ELOレーティング（プレイヤーには段位のみ表示）

**シーズン:**
- 期間: 1ヶ月（毎月1日リセット）
- リセット: ランクは一定段階まで降格（例：金3→銀5）
- シーズン報酬: 到達最高ランクに応じたコスメ（スリーブ・称号・フレーム）
- ランキング: シーズン終了時トップ100にボーナス報酬

**PvEコンテンツ方針（Phase 3〜4で導入）:**
- AI Botチャレンジ: 願主ごとの高難度AI戦（報酬付き）
- パズルバトル: 1ターンで条件を満たす詰め将棋形式
- 週替わりボスチャレンジ: 特殊ルールのAI戦（高HP願主・特殊界律）

**ストリーマー・観戦機能方針（Phase 3〜6で段階導入）:**
- Phase 3: リプレイ共有URL + 観戦モード（2ターン遅延）
- Phase 5: 観戦用ストリームオーバーレイ（手札隠蔽表示）
- Phase 6: トーナメント配信支援（自動カメラ切替・ハイライト検出）

### Phase 4: ストーリーと世界の拡張（MVP後 6〜9ヶ月）

目的: `万願果` の世界をゲームとして広げる

実装候補:
- ストーリー第2〜3章追加
- 願主ごとの個別エピソード
- ストーリーイベント（期間限定ストーリー + 報酬）
- 交界の地域・時代別設定追加
- キャラクター図鑑
- 用語集 / アーカイブ

### Phase 5: プラットフォーム拡張（MVP後 6〜12ヶ月）

目的: iOS以外のプレイヤーにリーチする

実装候補:
- **Steam版リリース**（広告なし、1.3xミッション報酬）
- Steam固有のUI調整（横持ち・キーボード対応）
- クロスプラットフォームアカウント連携
- クロスプレイ（iOS ↔ Steam）
- **Android版リリース**（Steamの後、iOS同等の広告+IAP）

### Phase 6: 大規模運営とeSports（MVP後 12ヶ月〜）

目的: 長期運営とコミュニティの成熟

実装候補:
- 公式大会・トーナメントモード
- コラボイベント基盤
- ギルド/チーム機能
- リプレイ共有SNS（コミュニティポータル）
- ハイライト自動生成
- 高度なコスメティック（フィールドスキン・ゲージスキン拡張）
- BigQuery連携の本格分析基盤
- ローカライゼーション（英語・繁体中文・韓国語）

## 4. 実装順のおすすめ

MVP後は以下の順番を推奨する。

1. バグ修正 + バランス初回パッチ
2. UI polish + サウンド強化 + アクセシビリティ P1
3. 第2弾シリーズ + 願主拡張
4. 構築制限イベント + 季節イベント
5. ランクマッチ + シーズン制
6. リプレイ + 観戦
7. ストーリー拡張
8. Steam版
9. Android版
10. 大規模運営機能

---

## 5. 先に作ると効果が高いもの

- 第2弾シリーズ（カードプール拡張はリテンションに直結）
- ランクマッチ（継続モチベーションの核）
- 構築制限イベント（既存カードプールで新鮮な体験を提供）
- Steam版（新規ユーザー獲得チャネル拡大）

---

## 6. 後回しでもよいもの

- コラボイベント（IP交渉に時間がかかる）
- ギルド/チーム機能
- ハイライト自動生成
- BigQuery連携
- 高度なローカライゼーション（5言語以上）

---

## 7. AI実装ルール

AIはポストMVP機能を作る際、以下を守ること。

- まず MVP の導線を壊さない
- 新機能は既存5画面のどこに入るかを明示する
- `GAME_DESIGN.md` と矛盾する新ルールを勝手に足さない
- 既存用語を崩さない
- 1機能ごとに依存関係を整理してから実装する
- 新カード追加時は `CARD_TEXT_GUIDELINES.md` に従う
- バランス調整時は `BALANCE_POLICY.md` の手順に従う
