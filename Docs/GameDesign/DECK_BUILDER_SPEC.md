# DECK_BUILDER_SPEC.md
> 万願果 デッキ構築UI/UX設計 v1.0

---

## 0. 目的

プレイヤーがカードを選んでデッキを構築するための画面設計・操作フロー・バリデーションを定義する。MONETIZATION_DESIGN.md v2.0 でパック・クラフトをMVPに含めたため、デッキ構築UIもMVP必須となる。

---

## 1. 構築ルール（GAME_DESIGN.md §6 準拠）

| ルール | 値 |
|--------|-----|
| デッキ枚数 | **34枚固定** |
| カードタイプ | 顕現・詠術・界律を自由に組み合わせ |
| 界律推奨上限 | 8枚（バランステスト次第で10枚まで拡張可） |
| 同名カード上限 | **3枚** |
| 願主 | デッキ外（デッキ選択時に願主も選択） |
| デッキ保存枠 | **初期5枠**（課金で最大15枠まで拡張可） |

### 2.2 デッキスロット表示

- 34枠を横一列で表示（横スクロール可）
- カード投入済みスロット: カードアイコン（縮小）+ コスト数字
- 空スロット: 暗色の空枠（破線）
- タップ → 該当カードの詳細表示 / 長押し → デッキから除外

### 2.3 マナカーブ（コスト分布グラフ）

- コスト0〜10+の棒グラフ
- 各コストにそのコストのカードが何枚あるか表示
- 棒の色はデッキ内で最も多い願相カラー
- 高さは最大枚数に対する比率

## 3. 操作フロー

### 3.1 カード追加

```
カード一覧でカードをタップ
  ├─ デッキが34枚未満 & 同名3枚未満
  │    → デッキスロットに追加（アニメーション: カードが上に飛ぶ）
  │    → カウンター更新
  │    → マナカーブ更新
  │    → 一覧側のカードに「×N」バッジ表示
  ├─ デッキが34枚
  │    → トースト「デッキが満杯です」+ shake
  └─ 同名3枚到達
       → トースト「同名カードは3枚までです」+ shake
```

### 3.2 カード除外

```
デッキスロットのカードをタップ
  → 確認なしで即除外（アニメーション: カードが下に戻る）
  → カウンター更新
  → マナカーブ更新

または: 長押し → 「除外」ボタン表示 → タップで除外
```

### 3.3 願主選択

```
願主ボタンをタップ
  → 願主一覧モーダル表示（MVP: 6願主）
  → 願主をタップで選択
  → 願主のKeyAspectとデッキ内願相の一致率を表示
```

### 3.4 デッキ保存

```
「保存」ボタンをタップ
  ├─ 34枚完成
  │    → 保存成功 + トースト「デッキを保存しました」
  ├─ 34枚未満
  │    → 確認ダイアログ「デッキが未完成です（N/34枚）。保存しますか？」
  │    → はい: 保存（使用不可マーク付き）/ いいえ: キャンセル
  └─ デッキ名未入力
       → デフォルト名で保存（「デッキ1」「デッキ2」…）
```

## 5. バリデーション

### 5.1 リアルタイムフィードバック

| 状態 | 表示 |
|------|------|
| 34枚未満 | カウンター赤色 + 「あとN枚」テキスト |
| 34枚完成 | カウンター緑色 + チェックマーク |
| 同名3枚到達 | 一覧側のカードにグレーアウト + 「MAX」バッジ |
| 界律8枚超え | 警告トースト「界律の推奨上限（8枚）を超えています」（投入は許可） |
| 未所持カード | シルエット表示 + 「生成」ボタン（クラフト画面へ遷移） |

### 5.2 デッキ使用条件

- 34枚未満のデッキは**対戦に使用不可**
- 保存は可能（編集途中として扱う）
- デッキ一覧で未完成デッキは「未完成」バッジ + グレーアウト

## 6. デッキコード（シェア機能）

### 6.1 エンコード

デッキをBase64テキストに圧縮する。

```
フォーマット: BNG1:{base64エンコード済みデータ}:{CRC8チェックサム(2桁hex)}
データ構造: [願主ID][カードID×枚数のペア配列]
例: BNG1:eyJ3IjoiV0lTSF9NQVNURVJfMDEiLCJjIjpbWyJNQU5fMDEiLDNdLFsiTUFOXzAyIiwyXV19:A3
```

**チェックサム（CRC-8）:**
- Base64データ部分のCRC-8を計算し、末尾に2桁hexで付加する
- インポート時にCRC-8を検証し、不一致の場合は「デッキコードが破損しています。コピーし直してください」と表示
- チェックサムなしの旧フォーマット（`BNG1:{base64}`）も後方互換で受け付ける（検証スキップ）

### 6.2 操作

| 操作 | フロー |
|------|--------|
| エクスポート | デッキ詳細 → 「コード」ボタン → クリップボードにコピー + トースト |
| インポート | デッキ一覧 → 「コードで作成」→ テキスト入力 → デッキプレビュー → 「この内容で作成」 |
| 未所持カード | インポート時に未所持カードがある場合、代替候補を提案 or そのまま作成（使用不可状態） |

### 6.3 SNSシェア

- デッキコード + デッキ名 + 願相分布のOGP画像を生成（MVP後）
- Twitter/Xシェアボタン（MVP後）

## 8. プリセットデッキ（MVP初期）

新規プレイヤーには以下のプリセットデッキを提供する。

| デッキ名 | 願主 | 中心願相 | 概要 | v0.5.3 戦略 |
|---------|------|---------|------|----------|
| **暁の衝撃** | 灯凪(曙) | 曙 | アグロ型。低コスト顕現 + Rush で攻撃的 | **高火力 HP ダメージで早期決着（鯱鉾勝利狙い）** |
| **蒼穹の策略** | Aldric(空) | 空 | コントロール型。妨害詠術 + 界律重視 | **ダメージ調整・WishTrigger 回避で塗り勝利へ** |
| **翠の連携** | 崔鋒(穏) | 穏 | ミッドレンジ型。展開 + 補助で盤面制圧 | **バランス型。HP 回復で長期戦対応** |

- プリセットデッキは編集不可（複製して編集する導線を提供）
- チュートリアル完了時に6願主から1つ選択 + 推奨デッキも所持
- **新規デッキ3種**: Amara(妖)コンボ型、Rahim(遊)テンポ型、崔鋒(玄)ディフェンス型（POST_MVP または季節ロテーション対応）

## 9. Firestore連携

### 9.1 データ構造

```
users/{uid}/decks/{deckId}/
  ├── name: string
  ├── wishMasterId: string
  ├── cards: { [cardId: string]: number }  // カードID→枚数
  ├── isComplete: boolean
  ├── createdAt: timestamp
  ├── updatedAt: timestamp
  └── isPreset: boolean
```

### 9.2 保存タイミング

- 「保存」ボタンタップ時に即Firestoreへ書き込み
- オフライン時はローカルキャッシュに保存、復帰時に同期
- 自動保存は行わない（意図しない変更を防ぐ）
